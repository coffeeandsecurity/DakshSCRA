DakshSCRA - Usage Guide

Source Code Review Assist — CLI & Web UI Reference

What is DakshSCRA?

DakshSCRA is a security-focused static analysis tool designed to support manual code review engagements. It surfaces areas of interest so security reviewers can focus attention where it matters most.

Purpose
DakshSCRA is a review assist tool, not an automated vulnerability scanner. Its findings represent areas requiring manual review, not confirmed vulnerabilities.
CapabilityDescription
Rule-based scanningMatches platform-specific patterns - SQL queries, command execution, file I/O, authentication constructs - against source files
ReconnaissanceDetects languages, frameworks, and technology stack from file contents and structure
Taint analysisTraces data flows from user-controlled sources to security-sensitive sinks across files and functions
Effort estimationEstimates manual review time based on codebase size, complexity, and finding volume
Report generationProduces single-file and per-platform HTML reports and optionally PDF

Installation

Requirements

  • Python 3.9 or later
  • pip
  • Playwright Chromium - only required for PDF generation

Setup

# Create and activate a virtual environment
python3.9 -m venv venv
source venv/bin/activate         # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Install Chromium (PDF generation only)
playwright install chromium

Docker - Web UI

docker compose up --build

Open http://localhost:8080 - change port with DAKSH_PORT=9090 docker compose up.

CLI Usage

Syntax

python dakshscra.py [SCAN OPTIONS] [MODE OPTIONS] [OUTPUT OPTIONS] [ADVANCED OPTIONS]

Run with no arguments to display the full help screen:

python dakshscra.py

Scan Options

OptionDescription
-r RULESPlatform rules to apply. Comma-separated (e.g. php,java) or auto for automatic detection.
-f FILE_TYPESOverride default file extensions for the selected platform.
-t TARGET_DIRPath to the source code directory to scan.
-v / -vv / -vvvVerbosity level. Higher levels show more detail during scanning.
Auto mode
Use -r auto to automatically detect languages and apply the appropriate rules. Recommended for unfamiliar codebases.

python dakshscra.py -r auto -t ./codebase

Mode Options

OptionDescription
--reconRun technology stack reconnaissance. Detects languages, frameworks, libraries. Can run alone or alongside -r.
--rs / --recon-strictFilter recon output to high-confidence detections only. Use with --recon to reduce noise.
--estimateGenerate a manual review effort estimation report.
--pdf-from-jsonGenerate PDF reports from existing JSON output without re-running a scan.
-l {R,RF}List available platform rules. R = rules only, RF = rules + file types.

Combining Modes

# Recon + scan + estimate in one pass
python dakshscra.py --recon --estimate -r php -t ./app

# Recon only - no scanning
python dakshscra.py --recon -t ./app

# Recon with strict confidence filter
python dakshscra.py --recon --rs -t ./app

Output Options

OptionDefaultDescription
-rpt FORMATShtmlReport formats: html, pdf, or html,pdf.
--json-input-dir PATH./reports/jsonJSON input directory for --pdf-from-json.
--pdf-output PATH./reports/pdf/report.pdfCustom path for the single combined PDF.
--pdf-multi-dir PATH./reports/pdf/multi-fileCustom directory for the per-platform PDF set.
--pdf-single-onlyoffGenerate only the combined single PDF; skip per-platform PDFs. Works in both scan and --pdf-from-json mode.
PDF not default
PDF is not generated by default. After any scan without PDF, the tool prints a reminder showing how to generate it from the saved JSON output.

Advanced Options

--locCount effective lines of code. Included in effort estimation output.

Understanding Findings

Scanner Findings

Each scanner finding represents a pattern match against a known-risky code construct. They are areas of interest - not confirmed vulnerabilities. Manual review is required to determine exploitability.

FieldDescription
RuleThe rule name (e.g. sql_injection, command_exec)
FileSource file path relative to the scan target
LineLine number of the match
SnippetSurrounding code context
PlatformLanguage/framework the rule belongs to
Common False Positives
Commented-out code, test fixtures, safe wrapper functions, and vendor library code all commonly match scanner rules. Always review context before treating a finding as a vulnerability.

Taint Analysis Findings

Taint findings are higher-signal than scanner findings because they trace actual data flow paths through the code.

FieldDescription
Confidencehigh cross-file, multiple steps   medium same-file, partial trace   low file-scope fallback
SourceWhere user-controlled data originates (request param, env var, file input, etc.)
SinkThe security-sensitive call that receives the tainted data
TraceOrdered list of intermediate steps connecting source to sink
Cross-fileWhether the flow spans more than one file - increases confidence

Confidence Interpretation

  • high - Cross-file flows with assignment steps. Highest priority for review.
  • medium - Same-file flows with partial tracing. Likely worth reviewing.
  • low - File-scope fallback or single-step match. May be noise; verify manually.

Recon Results

Recon output is categorised by technology type (Framework, Language, Frontend, Backend, etc.). Each detection shows the technology name, confidence level, and evidence files that triggered the detection.

Reducing Noise
Use --recon --rs to show only high/medium-confidence detections when a large or generic codebase generates excessive low-confidence matches.

RDL — Rule Description Language

RDL is DakshSCRA's rule logic layer for contextual filtering, suppression, and report rationale. In the current engine, RDL lives in external .rdl files and is attached to XML rules through <rdl_ref>. The older inline <rdl> form is retired.

Current Model
XML defines candidate matches and presentation. The external .rdl script decides whether a candidate stays visible, gets suppressed, and what reason/trace metadata is stored with that decision.

Rule Evaluation Sequence

  1. Reconnaissance or explicit CLI selection chooses the active platform/framework rule sets.
  2. The XML rule is loaded from rules/scanning/platform/....
  3. <regex> finds candidate matches.
  4. Optional <exclude> removes obvious noise for that rule.
  5. The external .rdl script from <rdl_ref> runs against the full current file, or against the normalized relative path for file-path rules.
  6. If the RDL script passes, the finding is reported. If it fails, the match is suppressed with the recorded fail reason and trace.

Rule XML Structure

<rule>
  <name>Rule Name</name>
  <regex><![CDATA[regex_to_match]]></regex>
  <rdl_ref>logic/common/core/insecure_sql_query_unsafe_string_concatenation.rdl</rdl_ref>
  <exclude><![CDATA[pattern_to_exclude_lines]]></exclude>  <!-- optional -->
  <scan_config>...</scan_config>                           <!-- optional -->
  <rule_desc>Short description of what the rule detects.</rule_desc>
  <vuln_desc>Why the pattern matters.</vuln_desc>
  <developer>Fix guidance for developers.</developer>
  <reviewer>Manual confirmation guidance for reviewers.</reviewer>
</rule>

<regex> finds candidates. <rdl_ref> loads the external logic script. scan_config controls how the match is scanned, aggregated, highlighted, and rendered in the report.

External RDL File Structure

VERSION 1
WHEN PRESENT /\b(?:mysql_query|mysqli_query|->query)\s*\(/i
WHEN EXPR PRESENT:\$_(GET|POST|REQUEST|COOKIE) && MISSING:\b(?:prepare|bindParam|bindValue|PDO::prepare)\b
REPORT AS area_of_interest
REASON Query execution appears to rely on direct input without parameterisation.
FAIL_REASON Matching query API was found, but prepared-statement indicators also exist in the same file.
TRACE SQLi gate: input source present and mitigation missing.

RDL is evaluated against the entire current file, not only the matched line. For file-path analysis, the current file text is the normalized relative path string.

RDL Commands

CommandBehaviourWhen to use
WHEN PRESENT <regex>Require a pattern to appear in the current file text.Confirm a risky API or sensitive field exists
WHEN MISSING <regex>Require a pattern to be absent from the current file text.Suppress when a mitigation is already present
WHEN EXPR <expr>Evaluate a boolean expression using PRESENT:, MISSING:, and EXISTS:.Combine multiple conditions compactly
WHEN CURRENT_FILE_MATCHES <regex>Match against the entire current file content.Whole-file config or structure checks
WHEN FILE_NAME_IS <name>Require an exact filename.AndroidManifest.xml, Info.plist, web.xml
WHEN FILE_PATH_MATCHES <glob>Require the current path to match a glob.Path-specific or framework-specific rules
UNLESS PRESENT <regex>Fail if a safe pattern is present.Early mitigation exclusion
UNLESS CURRENT_FILE_MATCHES <regex>Fail if the full file matches an exclusion pattern.Skip known-safe structural cases
OBSERVE PROJECT_HAS_GLOB <glob> AS <label>Record related files from the project root in trace metadata.Surface supporting config or companion files
REPORT AS <outcome>Set the rule outcome.Usually area_of_interest
REASON / FAIL_REASON / TRACEStore decision rationale and trace text.Reviewer context and suppression reasons

WHEN EXPR supports PRESENT:, MISSING:, EXISTS:, &&, ||, !, and parentheses.

Examples

Example 1 — PHP SQL injection gating

<rule>
  <name>Possible SQL Injection in Query Execution</name>
  <regex><![CDATA[(?i)\b(?:mysql_query|mysqli_query|->query)\s*\(]]></regex>
  <rdl_ref>logic/common/core/insecure_sql_query_unsafe_string_concatenation.rdl</rdl_ref>
</rule>

VERSION 1
WHEN PRESENT /\b(?:mysql_query|mysqli_query|->query)\s*\(/i
WHEN EXPR PRESENT:\$_(GET|POST|REQUEST|COOKIE) && MISSING:\b(?:prepare|bindParam|bindValue|PDO::prepare)\b
REPORT AS area_of_interest
REASON Query execution appears to rely on direct input without parameterisation.

The regex stays broad enough to catch candidate query APIs. The external RDL file then keeps only the cases where input sources are present and parameterisation appears absent.

Example 2 — Android manifest rule using filename and full-file checks

<rule>
  <name>Exported Components Without Permission</name>
  <regex><![CDATA[<(?P<component>activity|service|receiver|provider)\s[^>]*android:name="(?P<name>[^"]+)"[^>]*android:exported="true"[^>]*(?:/>|>)]]></regex>
  <rdl_ref>logic/mobile/android/core/exported_components.rdl</rdl_ref>
</rule>

VERSION 1
WHEN FILE_NAME_IS AndroidManifest.xml
WHEN CURRENT_FILE_MATCHES /android:exported\s*=\s*"true"/i
WHEN MISSING /android:permission\s*=\s*"/i
REPORT AS area_of_interest
REASON Exported component appears reachable without a permission guard.

This is the typical split of responsibilities: regex finds candidates, scan_config controls presentation, and rdl_ref confirms the wider file context before the finding survives.

Example 3 — File-path area-of-interest rule

<rule>
  <name>Admin Section File Path</name>
  <regex><![CDATA[(?i)(^|/)(admin|administrator|root)(/|$)]]></regex>
  <rdl_ref>logic/filepaths/core/admin_section.rdl</rdl_ref>
</rule>

VERSION 1
WHEN CURRENT_FILE_MATCHES /(^|\/)(admin|administrator|root)(\/|$)/i
UNLESS CURRENT_FILE_MATCHES /(^|\/)(tests?|docs?|samples?|examples?)(\/|$)/i
REPORT AS area_of_interest
REASON File path suggests privileged application functionality.
FAIL_REASON Path matched an excluded documentation or sample location.

For file-path analysis, the regex and the RDL script operate on the normalized relative path string rather than source code text.

Example 4 — Project observation rule

VERSION 1
WHEN FILE_NAME_IS AndroidManifest.xml
WHEN CURRENT_FILE_MATCHES /android:usesCleartextTraffic\s*=\s*"true"/i
OBSERVE PROJECT_HAS_GLOB **/network_security_config*.xml AS network_security_configs
REPORT AS area_of_interest
REASON Cleartext traffic is enabled; related network security config files were recorded for reviewer context.

OBSERVE currently records supporting project files in trace metadata. It does not by itself gate the rule result.

How RDL Reduces False Positives

Without RDL, every regex hit is reported regardless of context. RDL adds the file-aware pass that keeps the useful matches and suppresses the obvious false positives:

PatternWithout RDLWith RDL
getSharedPreferences()Flags every preference access (high FP rate)Only flags when sensitive keys AND no encryption present
loadUrl(someVar)Flags hardcoded safe URLs like about:blankOnly flags dynamic / interpolated URLs
Room.databaseBuilder()Flags DB setup calls — zero injection risk (100% FP)Replaced with @Query string interpolation pattern only
System.getenv("TOKEN")Flags as hardcoded secret (FP — it is a safe read)Suppressed by MISSING:System.getenv condition
viewModelScope.launch {}Flags every coroutine dispatch including benign UI commandsOnly flags when body contains sensitive ops without error handling
Whole-file limitation
WHEN PRESENT, WHEN MISSING, and WHEN EXPR are evaluated against the entire file, not the matched line only. If a mitigation appears anywhere in the file, all candidates in that file can be suppressed. That trade-off is intentional: lower noise at the cost of occasionally hiding an issue in an otherwise-safe file.

Authoring Guidance

  • Keep <regex> broad enough to catch candidates, then use RDL to filter context.
  • Use <rdl_ref> for all new rules. Do not add new inline <rdl> blocks.
  • Prefer WHEN PRESENT / WHEN MISSING for simple gates and WHEN EXPR only when the logic genuinely needs boolean composition.
  • Put report-facing intent in REASON and suppression-facing intent in FAIL_REASON.
  • For file-path rules, write regex and RDL conditions against normalized slash-separated relative paths.

scan_config — Rule Engine & Reporting

scan_config is an optional XML block inside any <rule> that controls how the engine scans for that rule and how matches appear in reports. Rules without <scan_config> behave exactly as before — it is fully additive and backward-compatible.

Why it exists
Without scan_config you can only choose what to match. With it you control how to match (line-by-line vs whole-file), how much context to show around a match, whether to collapse all hits in a file into one finding, what format to render (default, component list, secret list), and which parts to highlight in colour.

Full Schema

<scan_config>
  <!-- ── Matching behaviour ──────────────────────────────────────────── -->
  <!-- line  : match regex line-by-line (default)                        -->
  <!-- file  : apply regex to full file with MULTILINE|DOTALL flags      -->
  <match_mode>line</match_mode>

  <!-- ── Context lines ──────────────────────────────────────────────── -->
  <!-- Raw source lines shown above / below each match in the report.   -->
  <!-- Default: 0. Independent settings.                                -->
  <context_lines_before>0</context_lines_before>
  <context_lines_after>0</context_lines_after>

  <!-- ── Aggregation ────────────────────────────────────────────────── -->
  <!-- none : one finding entry per match line (default)                -->
  <!-- file : collapse all matches in a file into one finding entry     -->
  <aggregate>none</aggregate>

  <!-- ── Report format ──────────────────────────────────────────────── -->
  <!-- default        : standard finding card (default)                 -->
  <!-- component_list : render named capture groups as a component list -->
  <!-- secret_list    : render matches as a redacted secrets table      -->
  <report_format>default</report_format>

  <!-- ── Highlighting ───────────────────────────────────────────────── -->
  <highlight_enabled>true</highlight_enabled>

  <!-- highlight_target options:                                        -->
  <!--   match   : highlight the full regex match (default)            -->
  <!--   groups  : highlight named capture groups listed in highlight_groups -->
  <!--   pattern : highlight occurrences of highlight_pattern          -->
  <highlight_target>match</highlight_target>
  <highlight_groups></highlight_groups>    <!-- comma-separated group names -->
  <highlight_pattern></highlight_pattern>  <!-- regex used in pattern mode  -->

  <!-- Colour applied to highlighted spans (CLI ANSI + web CSS)         -->
  <!-- Palette: red  yellow  cyan  green  magenta  bold                 -->
  <highlight_color>red</highlight_color>

  <!-- ── Multi-pass marks (optional) ───────────────────────────────── -->
  <!-- Each <mark> is one independent highlight pass over the snippet.  -->
  <!-- Useful when different parts of a line need different colours.    -->
  <marks>
    <mark><pattern>regex_A</pattern><color>red</color></mark>
    <mark><pattern>regex_B</pattern><color>cyan</color></mark>
  </marks>
</scan_config>

Platform Profiles

Three canonical profiles cover the vast majority of rule types. Pick the closest one and adjust as needed.

Profile A — Structured / Manifest Files

Applies to YAML, XML manifests, HCL, .env files, and other config formats where each key-value is one line and all matches in a file should be collapsed into a single finding.

<!-- Example: Android cleartext traffic setting in AndroidManifest.xml -->
<rule>
  <name>Android Cleartext Traffic Allowed</name>
  <regex><![CDATA[android:usesCleartextTraffic\s*=\s*"true"]]></regex>
  <rdl_ref>logic/mobile/android/core/cleartext_traffic.rdl</rdl_ref>
  <scan_config>
    <match_mode>line</match_mode>
    <context_lines_before>2</context_lines_before>
    <aggregate>file</aggregate>
    <highlight_enabled>true</highlight_enabled>
    <highlight_color>red</highlight_color>
  </scan_config>
  <rule_desc>...</rule_desc>
</rule>

Profile B — Source Code (Injection / RCE)

Applies to dangerous sink calls in source files where surrounding lines are critical for understanding data flow. Show 3–4 lines of context; do not aggregate (each call site is an independent finding).

<!-- Example: PHP shell_exec with possible user input -->
<rule>
  <name>Command Injection: shell_exec</name>
  <regex><![CDATA[(shell_exec|exec|passthru|system)\(]]></regex>
  <rdl_ref>logic/php/core/command_injection_shell_exec_exec_passthru_system_popen_backtick.rdl</rdl_ref>
  <scan_config>
    <match_mode>line</match_mode>
    <context_lines_before>4</context_lines_before>
    <context_lines_after>2</context_lines_after>
    <aggregate>none</aggregate>
    <highlight_enabled>true</highlight_enabled>
    <highlight_color>red</highlight_color>
  </scan_config>
  <rule_desc>...</rule_desc>
</rule>

Profile C — Hardcoded Secrets

Applies to credential/key pattern rules. Aggregate per file so the report lists all secrets found in a file together; use secret_list format for a clean redacted table.

<!-- Example: hardcoded API key or token -->
<rule>
  <name>Hardcoded Secret Material</name>
  <regex><![CDATA[(?i)(api[_-]?key|secret|token|password)\s*[:=]\s*["'][^"']{8,}["']]]></regex>
  <rdl_ref>logic/common/core/server_side_secrets_handling.rdl</rdl_ref>
  <scan_config>
    <match_mode>line</match_mode>
    <context_lines_before>1</context_lines_before>
    <aggregate>file</aggregate>
    <report_format>secret_list</report_format>
    <highlight_enabled>true</highlight_enabled>
    <highlight_color>red</highlight_color>
  </scan_config>
  <rule_desc>...</rule_desc>
</rule>

Highlighting

Highlighting applies ANSI colour codes in CLI output and CSS <span> classes in HTML/web output. The colour palette maps to both:

Color valueIntended useExamples
redCritical / dangerous sinksRCE, SQLi, hardcoded secrets, TLS bypass
yellowWeak, deprecated, or configuration riskCleartext HTTP, debug flags, deprecated APIs
cyanStructural — component or permission namesExported component names, permission strings
greenMitigation identifiedhtmlentities, EncryptedSharedPreferences
magentaFramework-specific patterns needing reviewWebView JS bridge, InAppBrowser, eval in templates
boldNeutral emphasisFunction names, identifiers to call out

Simple highlight (full match)

<scan_config>
  <highlight_enabled>true</highlight_enabled>
  <highlight_target>match</highlight_target>   <!-- default -->
  <highlight_color>red</highlight_color>
</scan_config>

Named capture groups — component list

Use named groups in the regex and highlight_target:groups to highlight only the captured parts:

<regex><![CDATA[android:name\s*=\s*"(?P<component>[^"]+)".*android:exported\s*=\s*"true"]]></regex>
<scan_config>
  <report_format>component_list</report_format>
  <aggregate>file</aggregate>
  <highlight_target>groups</highlight_target>
  <highlight_groups>component</highlight_groups>
  <highlight_color>cyan</highlight_color>
</scan_config>

Multi-pass marks — two colours on one line

<scan_config>
  <marks>
    <mark><pattern>android:exported\s*=\s*"true"</pattern><color>red</color></mark>
    <mark><pattern>android:name\s*=\s*"[^"]+"</pattern><color>cyan</color></mark>
  </marks>
</scan_config>

Rule Authoring Guide

Answer these five questions to derive the correct scan_config for any new rule:

QuestionAnswerSetting
Q1. Does the pattern span multiple lines?Yes — multi-line blockmatch_mode: file
No — single linematch_mode: line
Q2. How critical is the finding?Critical (RCE, SQLi, SSRF, TLS bypass)context_lines_before: 3–4, color: red
Medium (weak config, deprecated API)context_lines_before: 1–2, color: yellow
Mitigation / structuralcolor: green or cyan
Q3. Does each match represent an independent finding?Yes — each call site matters individuallyaggregate: none
No — any occurrence in a file is the issueaggregate: file
Q4. What should the report look like?Standard finding cardreport_format: default
List of captured components (permissions, names)report_format: component_list
Redacted secrets tablereport_format: secret_list
Q5. What should be highlighted?Whole regex matchhighlight_target: match
Named capture groups onlyhighlight_target: groups + highlight_groups

Pre-submission checklist

Before committing a new rule
  • <scan_config> block is present (even if using all defaults)
  • match_mode is correct for the pattern type (single-line vs multi-line)
  • Context lines are set — at least context_lines_before: 1 for all non-trivial rules
  • Critical rules use color: red; weak/deprecated use yellow
  • Secret rules use report_format: secret_list and aggregate: file
  • Manifest/config rules use aggregate: file
  • XML is valid — run python3 -c "import xml.etree.ElementTree as ET; ET.parse('your_file.xml')"

PDF Reports

PDF generation requires Playwright's Chromium browser to be installed:

playwright install chromium

Generate PDF during a scan

python dakshscra.py -r php -t ./app -rpt html,pdf

Generate PDF after a scan

# Both single and per-platform PDFs
python dakshscra.py --pdf-from-json

# Single combined PDF only
python dakshscra.py --pdf-from-json --pdf-single-only

# From a custom JSON directory
python dakshscra.py --pdf-from-json --json-input-dir ./custom/reports/json
JSON is always saved
JSON output is saved regardless of the -rpt setting. PDF can always be generated later from the JSON without re-running the scan.

Tips & Common Patterns

First scan on an unknown codebase

python dakshscra.py -r auto --recon --estimate -t ./target

Detects the technology stack, applies the relevant rules, and produces an effort estimate - the recommended starting point for a new engagement.

Large codebases - fast initial pass

python dakshscra.py -r auto --skip-analysis -t ./large_app

Skipping taint analysis significantly reduces scan time. Run the full analysis separately once you have recon and scanner results to guide you.

Resuming an interrupted scan

python dakshscra.py --resume-scan -r php -t ./app

Scan state is checkpointed by default. Use --resume-scan to continue from the last saved point rather than starting over.

Suppressing known false positives

  1. Run the initial scan and review findings.
  2. Generate a baseline from current findings: python dakshscra.py -r php -t ./app --baseline-generate
  3. On subsequent scans the baseline is loaded automatically, suppressing known-benign findings from reports.
  4. Use --no-baseline to temporarily ignore the baseline if needed.

About

DakshSCRA was built as a structured source code review assistant for complex manual security reviews. It is designed to help reviewers move beyond ad hoc pattern searching and toward a more systematic, coverage-oriented review process across large codebases, multiple platforms, and diverse frameworks.

The project was started by Debasis Mohanty after many years of using a mix of manual review methods and custom scripting during real-world source code review engagements. The goal was not simply to build another pattern-matching scanner, but to create a flexible code review tool that could guide both reviewers and developers through a structured and holistic analysis workflow.

Why It Was Built

Debasis has been performing code reviews for more than two decades across a wide range of platforms, frameworks, and architectures. That experience made one point clear: source code review is tedious, mentally demanding, and often difficult to scale when an application becomes large, distributed, or highly customized.

Over time, experienced reviewers learn that the only reliable way to maintain depth and coverage is to structure the review and follow a systematic approach. DakshSCRA was created from that idea. The tool is intended to support disciplined review methodology by surfacing areas of interest, organizing evidence, and performing advanced analysis that helps reviewers focus on what matters most.

Another motivation was the gap in the tooling market. Commercial products that perform strong taint analysis with real control-flow and data-flow tracing are often expensive niche offerings. Many open-source alternatives either stop at pattern checks or require heavy customization while still leaving major coverage gaps. DakshSCRA was written to set a much higher standard for open-source code review tooling, while keeping the architecture flexible enough to support deeper and broader review use cases.

History

  • Black Hat USA 2022: DakshSCRA was first introduced privately during the secure code review training Source Code Review (Raise Your Bar Beyond Grep). It was shown to trainees as part of the review methodology and supporting workflow, after having been kept private for years.
  • Black Hat USA 2023 Arsenal: DakshSCRA made its official public debut as an open-source tool demonstration. This was the point at which the project moved from a private capability into a public release.
  • Public Release: After the Black Hat USA 2023 Arsenal debut, the project was made publicly available on GitHub so the wider community could use it, study it, and contribute to its continued evolution.

Project Direction

The long-term direction of DakshSCRA is to remain a structured and systematic code review companion rather than a shallow alert generator. It aims to guide both developers and reviewers with areas of interest, advanced analysis, extensible rule logic, wide platform and framework coverage, and reporting that helps explain why something matters.

A central design goal has always been flexibility. The architecture is meant to be extensible so the tool can be adapted to different review styles, technologies, and depth requirements without forcing users into a narrow scanner-only model.

The broader objective is straightforward: provide a serious open-source code review tool without the ridiculous licensing costs typically associated with advanced commercial offerings, while continuing to push distinctive capabilities that raise the bar for what open-source review tooling can do.
