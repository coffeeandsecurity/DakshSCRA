DakshSCRA - Usage Guide

Source Code Review Assist — CLI & Web UI Reference

What is DakshSCRA?

DakshSCRA is a security-focused static analysis tool designed to support manual code review engagements. It surfaces areas of interest so security reviewers can focus attention where it matters most.

Purpose
DakshSCRA is a review assist tool, not an automated vulnerability scanner. Its findings represent areas requiring manual review, not confirmed vulnerabilities.
CapabilityDescription
Rule-based scanningMatches platform-specific patterns - SQL queries, command execution, file I/O, authentication constructs - against source files
ReconnaissanceDetects languages, frameworks, and technology stack from file contents and structure
Taint analysisTraces data flows from user-controlled sources to security-sensitive sinks across files and functions
Effort estimationEstimates manual review time based on codebase size, complexity, and finding volume
Report generationProduces single-file and per-platform HTML reports and optionally PDF

Installation

Requirements

  • Python 3.9 or later
  • pip
  • Playwright Chromium - only required for PDF generation

Setup

# Create and activate a virtual environment
python3.9 -m venv venv
source venv/bin/activate         # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Install Chromium (PDF generation only)
playwright install chromium

Docker - Web UI

docker compose up --build

Open http://localhost:8080 - change port with DAKSH_PORT=9090 docker compose up.

CLI Usage

Syntax

python dakshscra.py [SCAN OPTIONS] [MODE OPTIONS] [OUTPUT OPTIONS] [ADVANCED OPTIONS]

Run with no arguments to display the full help screen:

python dakshscra.py

Scan Options

OptionDescription
-r RULESPlatform rules to apply. Comma-separated (e.g. php,java) or auto for automatic detection.
-f FILE_TYPESOverride default file extensions for the selected platform.
-t TARGET_DIRPath to the source code directory to scan.
-v / -vv / -vvvVerbosity level. Higher levels show more detail during scanning.
Auto mode
Use -r auto to automatically detect languages and apply the appropriate rules. Recommended for unfamiliar codebases.

python dakshscra.py -r auto -t ./codebase

Mode Options

OptionDescription
--reconRun technology stack reconnaissance. Detects languages, frameworks, libraries. Can run alone or alongside -r.
--rs / --recon-strictFilter recon output to high-confidence detections only. Use with --recon to reduce noise.
--estimateGenerate a manual review effort estimation report.
--pdf-from-jsonGenerate PDF reports from existing JSON output without re-running a scan.
-l {R,RF}List available platform rules. R = rules only, RF = rules + file types.

Combining Modes

# Recon + scan + estimate in one pass
python dakshscra.py --recon --estimate -r php -t ./app

# Recon only - no scanning
python dakshscra.py --recon -t ./app

# Recon with strict confidence filter
python dakshscra.py --recon --rs -t ./app

Output Options

OptionDefaultDescription
-rpt FORMATShtmlReport formats: html, pdf, or html,pdf.
--json-input-dir PATH./reports/jsonJSON input directory for --pdf-from-json.
--pdf-output PATH./reports/pdf/report.pdfCustom path for the single combined PDF.
--pdf-multi-dir PATH./reports/pdf/multi-fileCustom directory for the per-platform PDF set.
--pdf-single-onlyoffGenerate only the combined single PDF; skip per-platform PDFs. Works in both scan and --pdf-from-json mode.
PDF not default
PDF is not generated by default. After any scan without PDF, the tool prints a reminder showing how to generate it from the saved JSON output.

Advanced Options

OptionDescription
--skip-analysisDisable taint analysis for this run. Scanning and recon still run normally.
--locCount effective lines of code. Included in effort estimation output.
--baseline-file PATHPath to a suppression baseline JSON file.
--baseline-generateGenerate a new suppression baseline from current findings.
--no-baselineIgnore the baseline suppression file for this run.
--resume-scanResume a previously interrupted scan from its last checkpoint.
--review-config PATHApply a findings triage file (JSON). Previously reviewed FPs and suppressed findings are excluded from reports.
--state-file PATHCustom path for the scan state/checkpoint file.
--no-stateDisable scan state checkpointing for this run.
--stateForce enable scan state checkpointing for this run.

CLI Examples

# Basic PHP scan
python dakshscra.py -r php -t ./src

# Multi-platform scan with verbose output
python dakshscra.py -r php,java -vv -t /path/to/code

# Auto-detect platforms and scan
python dakshscra.py -r auto -t ./codebase

# Full scan: recon + estimate + HTML + PDF
python dakshscra.py --recon --estimate -r php -t ./app -rpt html,pdf

# Recon only
python dakshscra.py --recon -t ./api

# Recon with strict confidence filter
python dakshscra.py --recon --rs -t ./mobile_app

# Skip taint analysis for faster scan

# Generate PDF from previous scan output
python dakshscra.py --pdf-from-json
python dakshscra.py --pdf-from-json --pdf-single-only
python dakshscra.py --pdf-from-json --json-input-dir ./custom/reports/json

# Resume an interrupted scan
python dakshscra.py --resume-scan -r php -t ./app

# List available rules
python dakshscra.py -l R
python dakshscra.py -l RF

Understanding Findings

Scanner Findings

Each scanner finding represents a pattern match against a known-risky code construct. They are areas of interest - not confirmed vulnerabilities. Manual review is required to determine exploitability.

FieldDescription
RuleThe rule name (e.g. sql_injection, command_exec)
FileSource file path relative to the scan target
LineLine number of the match
SnippetSurrounding code context
PlatformLanguage/framework the rule belongs to
Common False Positives
Commented-out code, test fixtures, safe wrapper functions, and vendor library code all commonly match scanner rules. Always review context before treating a finding as a vulnerability.

Taint Analysis Findings

Taint findings are higher-signal than scanner findings because they trace actual data flow paths through the code.

FieldDescription
Confidencehigh cross-file, multiple steps   medium same-file, partial trace   low file-scope fallback
SourceWhere user-controlled data originates (request param, env var, file input, etc.)
SinkThe security-sensitive call that receives the tainted data
TraceOrdered list of intermediate steps connecting source to sink
Cross-fileWhether the flow spans more than one file - increases confidence

Confidence Interpretation

  • high - Cross-file flows with assignment steps. Highest priority for review.
  • medium - Same-file flows with partial tracing. Likely worth reviewing.
  • low - File-scope fallback or single-step match. May be noise; verify manually.

Recon Results

Recon output is categorised by technology type (Framework, Language, Frontend, Backend, etc.). Each detection shows the technology name, confidence level, and evidence files that triggered the detection.

Reducing Noise
Use --recon --rs to show only high/medium-confidence detections when a large or generic codebase generates excessive low-confidence matches.

RDL — Rule Description Language

RDL is DakshSCRA's second-pass conditional filter applied after a regex match. Every rule can include an optional <rdl> block that adds context-aware conditions, significantly reducing false positives without writing separate rules for every edge case.

World's First (Open Source)
Conditional rule logic - previously found only in commercial security scanners - now available in open source. Rules can express multi-condition logic using FLAG, IF, PRESENT, MISSING, and boolean operators.

RDL conditions are evaluated against the entire file content of each matched file, not just the matched line. If the condition is not satisfied, the match is suppressed before it reaches the report.

Operators

OperatorBehaviourWhen to use
FLAG:<pattern>Anchors the condition — the subject the rule is built around.Always the first clause in an RDL expression
IF(condition)Match is only reported when this condition evaluates to true.Wraps PRESENT / MISSING predicates
PRESENT:<pattern>True when the pattern is found anywhere in the file.Require a co-occurring risky call
MISSING:<pattern>True when the pattern is not found anywhere in the file.Suppress when a mitigation is already present
EXISTS:<pattern>Like PRESENT but evaluated at file-path level.Check for presence of a related config file
&&Both conditions must hold.Require multiple simultaneous conditions
||Either condition must hold.Match any one of several conditions
!Negation.Invert a predicate

Examples

Example 1 — PHP SQL injection with missing parameterisation

<!-- Regex matches any mysql_query / mysqli_query call -->
<regex><![CDATA[(?i)(?:mysql_query|mysqli_query|->query)\s*\(]]></regex>

<!-- RDL: only flag when user input present AND no prepared statements found -->
<rdl><![CDATA[[FLAG:\$_(GET|POST|REQUEST|COOKIE)][IF(MISSING:(?:prepare|bindParam|bindValue|PDO::prepare))]]></rdl>

Fires when user-controlled input ($_GET, $_POST, etc.) is present and no parameterised query methods are found in the file. A file already using PDO::prepare is not flagged.

Example 2 — Android SharedPreferences storing sensitive data without encryption

<rdl><![CDATA[[FLAG:getSharedPreferences\(]
  [IF(PRESENT:(token|secret|password|auth|session)
   && MISSING:(EncryptedSharedPreferences|MasterKey|KeyStore|Cipher|encrypt))]
]]></rdl>

Only fires when the file references sensitive field names (token, password, etc.) and no Android encryption APIs are present. Files using EncryptedSharedPreferences are automatically suppressed.

Example 3 — Hardcoded secrets excluding environment-variable reads

<rdl><![CDATA[[FLAG:(api_key|secret|token|password)\s*[:=]\s*"[^"]{8,}"]
  [IF(MISSING:System\.getenv\s*\(|System\.getProperty\s*\(|BuildConfig\.
   && MISSING:example|sample|dummy|test|placeholder)]
]]></rdl>

Without this RDL, TOKEN = "${System.getenv("TOKEN")}" would be flagged as a hardcoded secret. The MISSING conditions exclude reads from environment variables, build config, and placeholder values in tests or comments.

Full rule XML structure

<rule>
  <name>Rule Name</name>
  <regex><![CDATA[regex_to_match]]></regex>
  <rdl><![CDATA[[FLAG:anchor_pattern][IF(PRESENT:risky_pattern && MISSING:mitigation_pattern)]]></rdl>
  <exclude><![CDATA[pattern_to_exclude_lines]]></exclude>  <!-- optional -->
  <rule_desc>Short description.</rule_desc>
  <vuln_desc>Vulnerability class explanation.</vuln_desc>
  <developer>Fix guidance for developers.</developer>
  <reviewer>Confirmation guidance for reviewers.</reviewer>
</rule>

How RDL Reduces False Positives

Without RDL, every regex match is reported regardless of context — leading to high noise. RDL adds a second pass that checks the broader file context before a match is reported:

PatternWithout RDLWith RDL
getSharedPreferences()Flags every preference access (high FP rate)Only flags when sensitive keys AND no encryption present
loadUrl(someVar)Flags hardcoded safe URLs like about:blankOnly flags dynamic / interpolated URLs
Room.databaseBuilder()Flags DB setup calls — zero injection risk (100% FP)Replaced with @Query string interpolation pattern only
System.getenv("TOKEN")Flags as hardcoded secret (FP — it is a safe read)Suppressed by MISSING:System.getenv condition
viewModelScope.launch {}Flags every coroutine dispatch including benign UI commandsOnly flags when body contains sensitive ops without error handling
File-scope limitation
PRESENT and MISSING conditions are evaluated against the entire file, not per-line. If a mitigation pattern appears anywhere in the file, all matches in that file are suppressed — even an unprotected call in the same file. This is a deliberate trade-off: lower noise at the cost of occasionally missing an issue in an otherwise-safe file. The reviewer note on every finding always advises manual confirmation for this reason.

PDF Reports

PDF generation requires Playwright's Chromium browser to be installed:

playwright install chromium

Generate PDF during a scan

python dakshscra.py -r php -t ./app -rpt html,pdf

Generate PDF after a scan

# Both single and per-platform PDFs
python dakshscra.py --pdf-from-json

# Single combined PDF only
python dakshscra.py --pdf-from-json --pdf-single-only

# From a custom JSON directory
python dakshscra.py --pdf-from-json --json-input-dir ./custom/reports/json
JSON is always saved
JSON output is saved regardless of the -rpt setting. PDF can always be generated later from the JSON without re-running the scan.

Tips & Common Patterns

First scan on an unknown codebase

python dakshscra.py -r auto --recon --estimate -t ./target

Detects the technology stack, applies the relevant rules, and produces an effort estimate - the recommended starting point for a new engagement.

Large codebases - fast initial pass

python dakshscra.py -r auto --skip-analysis -t ./large_app

Skipping taint analysis significantly reduces scan time. Run the full analysis separately once you have recon and scanner results to guide you.

Resuming an interrupted scan

python dakshscra.py --resume-scan -r php -t ./app

Scan state is checkpointed by default. Use --resume-scan to continue from the last saved point rather than starting over.

Suppressing known false positives

  1. Run the initial scan and review findings.
  2. Generate a baseline from current findings: python dakshscra.py -r php -t ./app --baseline-generate
  3. On subsequent scans the baseline is loaded automatically, suppressing known-benign findings from reports.
  4. Use --no-baseline to temporarily ignore the baseline if needed.
